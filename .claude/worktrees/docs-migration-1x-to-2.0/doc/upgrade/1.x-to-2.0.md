# Upgrade guide: Sōzu v1.1.x → v2.0.0

This guide covers the breaking changes operators need to address when
upgrading from a Sōzu v1.1.x deployment to v2.0.0. v2.0.0 is a major
version bump because the post-1.1.1 work on `main` accumulated breaking
changes in the protobuf IPC schema, dashboard metric names, default
behaviours, and config-load validation.

> Status of this guide. **Structural draft (Week 1 of v2.0.0 cut).** A
> follow-up commit augments this guide with the §3.2.e 302/308 redirect
> proto details once they land in Week 2.

## Audience

Operators running `sozu` v1.1.x in production, including:

- Self-hosted deployments managed via `ansible-clever`,
  `terraform-clever-cloud`, or hand-rolled TOML;
- Hosted CleverCloud tenants (cleverapps.io shape);
- Embedders consuming `sozu-command-lib` (`sozu-client`,
  `sozu-prometheus-connector`, `loadbalancer-agent`, `clever-operator`).

## Pre-upgrade checklist (mandatory)

Run these BEFORE swapping the v2.0.0 binary in. Without these backups, the
[rollback procedure](#rollback-procedure) cannot fully restore v1.1.x
state.

```sh
# 1. Snapshot the live state into a JSON backup
sozuctl state save /var/lib/sozu/state.v1.1.x.backup.json

# 2. Snapshot the audit log directory (the v2.0.0 JSON sink schema is not
#    backwards-compatible with v1.1.x consumers)
cp -a /var/log/sozu/audit/ /var/log/sozu/audit.v1.1.x.backup/

# 3. Record the running version
sozu --version > /var/lib/sozu/version.v1.1.x.txt

# 4. Snapshot Grafana / Prometheus dashboards externally — v2.0.0 metric
#    renames will break v1.1.x dashboards. Save the JSON exports for
#    rollback if needed.
```

## Breaking changes in v2.0.0

### 1. Proto-tag renumbering for health-check carriers ([#1191](https://github.com/sozu-proxy/sozu/pull/1191))

The post-PR-#1209 rebase shifted the proto-field tags PR #1191 originally
used. **External clients that hard-coded the older tags must be rebuilt.**
The new tags are:

| Field | v1.1.x tag | v2.0.0 tag |
|---|---|---|
| `Request.set_health_check` | 47 | **52** |
| `Request.remove_health_check` | 48 | **53** |
| `Request.query_health_checks` | 49 | **54** |
| `Cluster.health_check` | 8 | **15** |
| `ResponseContent.health_checks_list` | 14 | **15** |
| `EventKind.HEALTH_CHECK_HEALTHY` | 5 | **27** |
| `EventKind.HEALTH_CHECK_UNHEALTHY` | 4 | **28** |

**Action**: rebuild `sozu-client`, `sozu-prometheus-connector`,
`loadbalancer-agent`, `clever-operator`, and any in-house tooling against
the v2.0.0 `sozu-command-lib`. Mixed-version IPC during the brief upgrade
window relies on prost forward-compat (unknown-tag preservation), but the
sooner clients bump, the better.

### 2. Dashboard metric renames ([#1196](https://github.com/sozu-proxy/sozu/pull/1196))

The following metric names changed. Dashboards scraping the old names
will go blank until updated.

| v1.1.x | v2.0.0 |
|---|---|
| `client.connections_percentage` | `client.connections_percent` |
| `client.max_connections` | `client.connections_max` |
| `buffer.number` | `buffer.in_use` |

The high-resolution `client.connections` signal is preserved. Per-event
emission of `client.connections_*` from `SessionManager::incr/decr` is
dropped — the new names emit from the run loop alongside the proxy
gauges.

**Action**: update Grafana dashboards, statsd → Prometheus exporters, and
any `metrics get` scripts that reference the old names. Snapshot existing
dashboards before upgrade so rollback restores them.

### 3. `http.errors` now labelled with `(cluster_id, backend_id)` ([#1196](https://github.com/sozu-proxy/sozu/pull/1196))

`http.errors` is now emitted with two labels and filtered centrally by
the `metrics.detail` knob:

- `metrics.detail = "process"` (legacy default before opt-in landed) →
  collapses to a proxy-wide counter (no behaviour change for default
  v1.1.x deployments).
- `metrics.detail = "frontend"` → also collapses (no per-cluster split).
- `metrics.detail = "cluster"` (the documented default) → per-cluster
  attribution.
- `metrics.detail = "backend"` → per-cluster + per-backend split.

**Action**: dashboards that grouped `http.errors` by frontend or backend
will now see the labelled emission. If your statsd / Prometheus drain
auto-strips labels into the metric name, you may see new
`http.errors{cluster_id=…,backend_id=…}` series. Confirm
`metrics.detail` matches your operator intent.

### 4. Pattern-trie segment regex anchoring (CWE-1023 fix)

Every segment-level regex inserted into the routing trie is now wrapped
with `\A...\z` at insert time. A pattern like `cdn[0-9]+` now matches
`cdn1` exactly but **not** `cdn1xxx`.

**Operator impact**: regex frontends that relied on partial-segment
matching (e.g. `cdn[0-9]+` rule inadvertently matching `cdn123xxx`) will
break — traffic that used to be matched no longer is.

**Action**: audit your regex frontends BEFORE upgrade. v2.0.0 ships an
`sozu config diagnose-regex` CLI verb (per §7 of the v2.x plan, lands in
the small-wins cluster) that scans the active state and reports patterns
that would match more hostnames in v1.1.x than in v2.0.0. If you cannot
audit pre-upgrade, widen patterns explicitly: `cdn[0-9]+.*` to keep
v1.1.x semantics.

### 5. HAProxy-style empty-`val` del-header semantic

`Header { position, key, val }` entries are now actioned with HAProxy
parity:

- `position = SET, val = "x"` → set the header to `"x"`.
- `position = APPEND, val = "x"` → append a header with value `"x"`.
- `position = SET, val = ""` → **DELETE the header by name**
  (HAProxy `del-header` parity).

**Operator impact**: any v1.1.x configuration that set custom headers
to the empty string (intending "no value, but present") will now have
those headers DELETED from forwarded requests.

**Action**: audit your TOML / runtime-API state for `Header { val = "" }`
entries. If you intend to keep the header present, set a whitespace-only
value `" "` instead — though that is malformed per RFC 9110 §5.5 and
should be reviewed. v2.0.0 ships a `sozu config audit-empty-headers` CLI
verb (per §7 small-wins) to enumerate affected frontends.

### 6. Sub-16 393 `buffer_size` rejected at config load

When any `https_listeners[].alpn_protocols` includes `"h2"`, the global
`buffer_size` is now validated against `H2_MIN_BUFFER_SIZE = 16 393` at
boot (RFC 9113 §6.5.2 + §4.1: 16 384-byte max H2 frame payload + 9-byte
frame header).

A v1.1.x deployment with `buffer_size = 8192` and any H2 listener will
**refuse to start** v2.0.0 with `ConfigError::BufferSizeTooSmallForH2 {
buffer_size, minimum, listeners }`.

**Action**: either raise `buffer_size` to ≥ 16 393, or remove `"h2"`
from the affected listeners' `alpn_protocols`. The v1.1.x silent-deadlock
behaviour (where `buffer_size = 8192` started cleanly and only hung on
H2 frames under traffic) is no longer reachable.

### 7. Cipher-list constant rename ([#1191](https://github.com/sozu-proxy/sozu/pull/1191))

The internal Rust constant `DEFAULT_RUSTLS_CIPHER_LIST` in
`command/src/config.rs` is renamed to `DEFAULT_CIPHER_LIST`.

- **TOML key unchanged**: operators using `cipher_list = [...]` in TOML
  are NOT affected.
- **`sozu-command-lib` consumers** that imported the public constant by
  name must update.

The latent `DEFAULT_CIPHER_SUITES` (OpenSSL-style names that rustls
never matched) is dropped. `P-521` is removed from the default
`groups_list` (rustls 0.23 does not support it).

**Action**: rebuild downstream Rust crates that referenced the old
constant.

### 8. New TOML keys (additive; ignored on rollback)

v2.0.0 introduces several new TOML keys. They default to safe values; v1.1.x
silently ignores them on rollback.

| Key | Default | Meaning |
|---|---|---|
| `forwarded_for_mode` | `"x-forwarded"` | Per-frontend `Forwarded` header policy (`x-forwarded` / `rfc7239` / `both` / `none`). RFC 7239 support lands in v2.1.0. |
| `evict_on_queue_full` | `false` | Evict 1% LRU sessions when accept queue saturates instead of refusing new accepts. Defaults off (DDoS-safe). |
| `max_connections_per_ip` | `0` | Cap simultaneous frontend connections one source IP holds against a cluster. `0` = disabled. |
| `splice_pipe_capacity_bytes` | `None` | Linux `splice(2)` kernel-pipe size. `None` keeps kernel default 64 KiB. |
| `basic_auth_max_credential_bytes` | `4096` | Max decoded `Authorization: Basic` credential length. |
| `command_allowed_uids` | `[]` (any UID with mode-0600 access) | Optional `Vec<u32>` UID allowlist on the command socket. |
| `audit_logs_target` | `None` | Optional dedicated text audit log sink (PCI-DSS 10.5). |
| `audit_logs_json_target` | `None` | Optional dedicated JSON audit log sink for SIEM ingest. |

**Operator impact**: v2.0.0 starts with v1.1.x-equivalent behaviour
(eviction off, per-IP limit off, X-Real-IP elision off). Opt into the
new behaviours explicitly.

## Rollback procedure

If a regression surfaces post-v2.0.0 in production, rollback to v1.1.x:

```sh
# 1. Stop the v2.0.0 binary (graceful)
systemctl stop sozu

# 2. Restore the v1.1.x binary
install -m 0755 /var/lib/sozu/sozu.v1.1.1 /usr/bin/sozu

# 3. Restore the v1.1.x state file
cp /var/lib/sozu/state.v1.1.x.backup.json /var/lib/sozu/state.json

# 4. Revert TOML to v1.1.x shape (see warnings below)

# 5. Restore Grafana / Prometheus dashboards from your v1.1.x JSON snapshot

# 6. Start
systemctl start sozu
```

### Known not-rollback-able

- **State files written by v2.0.0 cannot be loaded by v1.1.x.** `SaveState`
  is JSON-serialised `WorkerRequest` with serde-generated names from the
  current proto schema. There is no proven forward-compat contract for
  older serde-generated structs. **Any state mutations after the v2.0.0
  upgrade (`sozu cluster add`, `sozu certificate add`, etc.) will NOT
  survive a binary rollback.** Manually replay post-upgrade changes
  against the pre-upgrade state file backup.
- **Audit log files** written by v2.0.0's JSON sink will not parse in
  v1.1.x consumers (new schema with audit-event v2 fields).
- **Access logs** with new `tls_version` / `tls_cipher` / `tls_sni` /
  `tls_alpn` / `xff_chain` fields will not parse in v1.1.x consumers.
- **Pattern-trie regex anchoring is a permanent code-level behaviour
  change.** Operators who relied on partial matching adjusted their
  regexes for v2.0.0 (per §4 above); rolling back to v1.1.x restores
  partial matching but operators have already widened patterns —
  net-neutral.
- **TOML mismatch hazard**: if you added H2 ALPN to a listener at
  upgrade time WITH `buffer_size < 16393`, v1.1.x WILL NOT REJECT the
  config (it just hangs on the first H2 frame). **Revert TOML changes
  at rollback.**
- **New TOML keys** (`forwarded_for_mode`, `evict_on_queue_full`,
  `max_connections_per_ip`, `splice_pipe_capacity_bytes`,
  `basic_auth_max_credential_bytes`, `command_allowed_uids`,
  `audit_logs_target`, `audit_logs_json_target`) are silently ignored
  by v1.1.x — remove them from TOML at rollback for clean parsing.

## Supported versions after v2.0.0

Per `SECURITY.md`:

- **`2.0.x`** — current; receives security fixes.
- **`1.1.x`** — **end-of-life at v2.0.0 release.** No further security
  backports. Operators are expected to upgrade to `2.x`.
- **`1.0.x`** and earlier — unsupported.
