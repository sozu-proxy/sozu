# Upgrade guide: Sōzu 1.1.x → 2.0.0

This guide covers the breaking changes operators need to address when
upgrading from a Sōzu 1.1.x deployment to 2.0.0. 2.0.0 is a major
version bump because the post-1.1.1 work on `main` accumulated breaking
changes in dashboard metric names, default behaviours, and config-load
validation.

## Audience

Operators running `sozu` 1.1.x in production, including:

- Self-hosted deployments managed via configuration-management tooling
  or hand-rolled TOML;
- Multi-tenant deployments;
- Embedders consuming `sozu-command-lib` from their own crates.

## How sōzu upgrades — two distinct paths

Sōzu has two `Command::new(...).exec()` sites that re-launch the binary,
each with a different intent. Operators should understand both before
running an upgrade.

### Master hot-upgrade — `sozuctl upgrade-main`

This is the **operator-triggered** path:

1. Operator installs the new binary at the running master's path
   (typically `/usr/bin/sozu`).
2. Operator runs `sozuctl --config <cfg> upgrade-main`.
3. The master forks; the child re-execs the **new** on-disk binary by
   path (`Command::new(executable_path)`). The whole point is to
   switch versions — the path-based exec is the intended behaviour.
4. The new master inherits listener fds + cluster state via SCM_RIGHTS
   over a temporary upgrade fd, then signals the old master to
   `soft_stop`. Old workers drain in-flight requests and exit.

### Worker auto-restart — internal to the master

This is the **non-interactive** path: when a worker panics or its
session count exceeds `max_connections`, the master spawns a
replacement.

Here the spawn must match the running master's version (otherwise the
master at version A and a fresh worker at version B exchange messages
under different proto schemas and crash). Until 2.0.0 the master used
the same `Command::new(executable_path).exec()` shape as the
hot-upgrade path — meaning a package upgrade that replaced the on-disk
binary between master startup and a worker auto-restart silently
spawned a worker at the **new** version, racing against the running
master at the old version. This is [#515].

The 2.0.0 fix opens `/proc/self/exe` with `O_PATH` **in the forked
child immediately before `execve(2)`** and uses `/proc/self/fd/<n>` for
the worker exec — kernel resolves through the magic symlink to the
**original** inode the master was started from, regardless of any
on-disk replacement. The forked child inherits the master's mmap'd
image, so `/proc/self/exe` in the child resolves to the same original
inode the master started from. Workers spawned via the auto-restart
path are now race-free against package upgrades.

The hot-upgrade path is **deliberately unchanged**: an operator running
`sozuctl upgrade-main` after `cp new-sozu /usr/bin/sozu` still gets the
new binary running the new master, exactly as before. The split is
documented in the rustdoc on `bin/src/util.rs::get_executable_exec_path()`.

#### Fallback when `/proc/self/exe` cannot be opened with `O_PATH`

The race-free worker exec path requires `/proc/self/exe` to be openable
with `O_PATH`. Operators running Sōzu in containers with masked `/proc`,
AppArmor / SELinux profiles denying `r /proc/self/exe`, or other
namespaced environments will see the worker fall back to path-based
exec (logged at `error!` level on every fallback). In that fallback
mode the #515 race window is unchanged from 1.1.x. To verify, grep
your worker logs for the string `falling back to path-based exec`.

#### Race-freedom assertion

Race-freedom under binary replacement is asserted by the kernel-tested
magic-symlink semantics of `/proc/self/exe`; the in-tree unit tests in
`bin/src/util.rs::tests` cover the helper's shape and fd uniqueness.

[#515]: https://github.com/sozu-proxy/sozu/issues/515

## Pre-upgrade checklist (mandatory)

Run these BEFORE swapping the 2.0.0 binary in. Without these backups, the
[rollback procedure](#rollback-procedure) cannot fully restore 1.1.x
state.

```sh
# 1. Snapshot the live state into a JSON backup
sozuctl state save /var/lib/sozu/state.1.1.x.backup.json

# 2. Snapshot the audit log directory (the 2.0.0 JSON sink schema is not
#    backwards-compatible with 1.1.x consumers)
cp -a /var/log/sozu/audit/ /var/log/sozu/audit.1.1.x.backup/

# 3. Record the running version
sozu --version > /var/lib/sozu/version.1.1.x.txt

# 4. Snapshot Grafana / Prometheus dashboards externally — 2.0.0 metric
#    renames will break 1.1.x dashboards. Save the JSON exports for
#    rollback if needed.
```

## Breaking changes in 2.0.0

### Dashboard metric renames

The following metric names changed. Dashboards scraping the old names
will go blank until updated.

| 1.1.x                           | 2.0.0                        |
| ------------------------------- | ---------------------------- |
| `client.connections_percentage` | `client.connections_percent` |
| `client.max_connections`        | `client.connections_max`     |
| `buffer.number`                 | `buffer.in_use`              |

The high-resolution `client.connections` signal is preserved. Per-event
emission of `client.connections_*` from `SessionManager::incr/decr` is
dropped — the new names emit from the run loop alongside the proxy
gauges.

**Action**: update Grafana dashboards, statsd → Prometheus exporters,
and any `metrics get` scripts that reference the old names. Snapshot
existing dashboards before upgrade so rollback restores them.

### `http.errors` is now labelled with `(cluster_id, backend_id)`

`http.errors` is now emitted with two labels and filtered centrally by
the `metrics.detail` knob:

- `metrics.detail = "process"` (legacy default before opt-in landed) →
  collapses to a proxy-wide counter (no behaviour change for default
  1.1.x deployments).
- `metrics.detail = "frontend"` → also collapses (no per-cluster split).
- `metrics.detail = "cluster"` (the documented default) → per-cluster
  attribution.
- `metrics.detail = "backend"` → per-cluster + per-backend split.

**Action**: dashboards that grouped `http.errors` by frontend or backend
will now see the labelled emission. If your statsd / Prometheus drain
auto-strips labels into the metric name, you may see new
`http.errors{cluster_id=…,backend_id=…}` series. Confirm
`metrics.detail` matches your operator intent.

### Pattern-trie segment regex anchoring (CWE-1023 fix)

Every segment-level regex inserted into the routing trie is now wrapped
with `\A...\z` at insert time. A pattern like `cdn[0-9]+` now matches
`cdn1` exactly but **not** `cdn1xxx`.

**Operator impact**: regex frontends that relied on partial-segment
matching (e.g. `cdn[0-9]+` rule inadvertently matching `cdn123xxx`) will
break — traffic that used to be matched no longer is.

**Action**: audit your regex frontends BEFORE upgrade. If you cannot
audit pre-upgrade, widen patterns explicitly: `cdn[0-9]+.*` to keep
1.1.x semantics.

### HAProxy-style empty-`val` del-header semantic

`Header { position, key, val }` entries are now actioned with HAProxy
parity:

- `position = SET, val = "x"` → set the header to `"x"`.
- `position = APPEND, val = "x"` → append a header with value `"x"`.
- `position = SET, val = ""` → **DELETE the header by name**
  (HAProxy `del-header` parity).

**Operator impact**: any 1.1.x configuration that set custom headers
to the empty string (intending "no value, but present") will now have
those headers DELETED from forwarded requests.

**Action**: audit your TOML / runtime-API state for `Header { val = "" }`
entries. If you intend to keep the header present, set a whitespace-only
value `" "` instead — though that is malformed per RFC 9110 §5.5 and
should be reviewed.

### Sub-16 393 `buffer_size` rejected at config load

When any `https_listeners[].alpn_protocols` includes `"h2"`, the global
`buffer_size` is now validated against `H2_MIN_BUFFER_SIZE = 16 393` at
boot (RFC 9113 §6.5.2 + §4.1: 16 384-byte max H2 frame payload + 9-byte
frame header).

A 1.1.x deployment with `buffer_size = 8192` and any H2 listener will
**refuse to start** 2.0.0 with `ConfigError::BufferSizeTooSmallForH2 {
buffer_size, minimum, listeners }`.

**Action**: either raise `buffer_size` to ≥ 16 393, or remove `"h2"`
from the affected listeners' `alpn_protocols`. The 1.1.x silent-deadlock
behaviour (where `buffer_size = 8192` started cleanly and only hung on
H2 frames under traffic) is no longer reachable.

### Cipher-list constant rename

The internal Rust constant `DEFAULT_RUSTLS_CIPHER_LIST` in
`command/src/config.rs` is renamed to `DEFAULT_CIPHER_LIST`.

- **TOML key unchanged**: operators using `cipher_list = [...]` in TOML
  are NOT affected.
- **`sozu-command-lib` consumers** that imported the public constant by
  name must update.

The latent `DEFAULT_CIPHER_SUITES` (OpenSSL-style names that rustls
never matched) is dropped. `P-521` is removed from the default
`groups_list` (rustls 0.23 does not support it).

**Action**: rebuild downstream Rust crates that referenced the old
constant.

### New TOML keys (additive; ignored on rollback)

2.0.0 introduces several new TOML keys. They default to safe values; 1.1.x
silently ignores them on rollback.

| Key                               | Default                              | Meaning                                                                                                                              |
| --------------------------------- | ------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------ |
| `forwarded_for_mode`              | `"x-forwarded"`                      | Per-frontend `Forwarded` header policy (`x-forwarded` / `rfc7239` / `both` / `none`). RFC 7239 support lands in a follow-up release. |
| `evict_on_queue_full`             | `false`                              | Evict 1% LRU sessions when accept queue saturates instead of refusing new accepts. Defaults off (DDoS-safe).                         |
| `max_connections_per_ip`          | `0`                                  | Cap simultaneous frontend connections one source IP holds against a cluster. `0` = disabled.                                         |
| `splice_pipe_capacity_bytes`      | `None`                               | Linux `splice(2)` kernel-pipe size. `None` keeps kernel default 64 KiB.                                                              |
| `basic_auth_max_credential_bytes` | `4096`                               | Max decoded `Authorization: Basic` credential length.                                                                                |
| `command_allowed_uids`            | `[]` (any UID with mode-0600 access) | Optional `Vec<u32>` UID allowlist on the command socket.                                                                             |
| `audit_logs_target`               | `None`                               | Optional dedicated text audit log sink (PCI-DSS 10.5).                                                                               |
| `audit_logs_json_target`          | `None`                               | Optional dedicated JSON audit log sink for SIEM ingest.                                                                              |

**Operator impact**: 2.0.0 starts with 1.1.x-equivalent behaviour
(eviction off, per-IP limit off, X-Real-IP elision off). Opt into the
new behaviours explicitly.

### systemd unit file: `Type=simple` → `Type=notify` (#228)

The bundled `os-build/systemd/sozu.service` and `sozu@.service` units
flip from `Type=simple` to `Type=notify` + `NotifyAccess=main`. The
master process now sends `READY=1` to systemd via `$NOTIFY_SOCKET`
only AFTER its initial workers have spawned and any saved state has
been replayed (see `command/src/sd_notify.rs`). `STOPPING=1` fires on
graceful shutdown; the hot-upgrade re-exec path also sends
`MAINPID=<new-pid>` followed by `READY=1`.

**Operator impact for the bundled units**: none — re-installing the
2.0.0 RPM / Arch / Debian package picks up the new unit file. After
`systemctl daemon-reload` the existing `systemctl status sozu` output
gains an `active (running)` state that flips only once the master is
truly ready; `After=sozu.service` peers (e.g. monitoring agents) can
now rely on that.

**Operator impact for custom unit files**: if you ship your own unit
file overriding `Type=simple`, you can switch to `Type=notify` +
`NotifyAccess=main` to get the readiness gate. The 2.0.0 binary is
also backwards-compatible with `Type=simple`: the helper is a no-op
when `$NOTIFY_SOCKET` is unset.

**Abstract `$NOTIFY_SOCKET` (containers, namespace-gated hosts):**
addresses starting with `@` (Linux-abstract sockets per `sd_notify(3)`)
are honoured. `command/src/sd_notify.rs::send_abstract` routes through
`AF_UNIX` + leading-NUL `sun_path` instead of resolving the value as
a filesystem path.

**`RELOADING=1` window around mutation verbs**: the master emits
`RELOADING=1` on entry and `READY=1` on completion of every mutating
verb dispatched through the command socket (LoadState, ReloadConfiguration,
AddCluster / Backend / Certificate / Frontend / Listener, Remove*,
Replace*, Update*, SetHealthCheck, SetMaxConnectionsPerIp, UpgradeMain,
UpgradeWorker, SoftStop / HardStop). Read-only verbs (Status, List*,
Query*, CountRequests, SubscribeEvents) skip the bracketing. Tooling
that watches `systemctl show -p ActiveState sozu` can serialise against
the reload window. The hot-upgrade path takes advantage of this gate:
the old master's outgoing `STOPPING=1` is suppressed so it cannot race
the new master's `MAINPID=` notify (which would otherwise flip the
unit to inactive while sozu is still serving traffic).

**`WatchdogSec=` is intentionally not set**: the bundled units do not
declare `WatchdogSec=`, and the master event loop does not yet emit
`WATCHDOG=1` periodic pings. The constant `STATE_WATCHDOG` is reserved
for the periodic-timer follow-up. **Operators adding `WatchdogSec=`
to a custom unit will see systemd kill the unit for missing pings —
do not enable it until the periodic-timer wiring lands.**

### TLS certificate chain dedup ([#1135](https://github.com/sozu-proxy/sozu/issues/1135) / [#1148](https://github.com/sozu-proxy/sozu/issues/1148))

ACME clients (Certbot's default `fullchain.pem`, lego, acme.sh) emit
the leaf certificate at the START of the chain file. In 1.1.x the
resolver stored the chain as `[leaf, leaf, intermediate, root]`
(double leaf), accepted by browsers but rejected by stricter
validators (Node.js → `UNABLE_TO_VERIFY_LEAF_SIGNATURE`).

In 2.0.0 the resolver dedups: each chain entry's DER bytes are
compared against the leaf and dropped on match. Each chain entry is
also fanned out through `split_certificate_chain` so a single string
carrying multiple PEM blocks (operators concatenating `fullchain.pem`
contents into one chain entry, common via the runtime API path) is
correctly split into one entry per CA before parsing.

**Operator impact**: none for correct chains. If your chain file was
already a `fullchain.pem` (leaf-first), the on-wire chain becomes
shorter by one cert; clients see a clean `[leaf, intermediate, …]`
chain. A `debug!` log line fires once per cert add when at least one
duplicate is dropped:

```
TLS-RESOLVER >>> dropped 1 duplicate leaf certificate(s) from the supplied chain
```

This is informational, not a warning — the dedup is the intended
behaviour. Operators wanting to spot it in production should set
`log_level = "debug"` for the `TLS-RESOLVER` target or watch the
on-wire chain length directly.

### Additional `RedirectPolicy` variants: 302 + 308 ([#1009](https://github.com/sozu-proxy/sozu/issues/1009))

The `RedirectPolicy` enum gains two variants alongside the existing
`PERMANENT` (301):

| Policy               | Status | RFC          | Method preserved on follow? |
| -------------------- | ------ | ------------ | --------------------------- |
| `PERMANENT`          | 301    | 9110 §15.4.2 | UA may rewrite POST → GET   |
| `FOUND`              | 302    | 9110 §15.4.3 | UA may rewrite POST → GET   |
| `PERMANENT_REDIRECT` | 308    | 9110 §15.4.9 | MUST preserve method        |

**Operator impact**: additive — existing frontends declared with
`redirect = permanent` continue to emit 301. Use `redirect = found`
or `redirect = permanent_redirect` only when the new semantics are
desired. Two new metrics ship: `http.302.redirection` and
`http.308.redirection` (cluster + backend labelled, mirroring the
existing `http.301.redirection`).

### Renamed inline-template compile-error counter

Inline-template compile failures previously surfaced under the
per-status counter `http.301.redirect_template.compile_error`. Now that
302 and 308 templates can fail the same way, the per-status form would
leave those failures unmonitored — so the counter is collapsed into a
single status-agnostic key:

| Before (1.1.x)                              | After (2.0.0)                           |
| ------------------------------------------- | --------------------------------------- |
| `http.301.redirect_template.compile_error`  | `http.redirect_template.compile_error`  |

**Operator impact**: dashboards or alerts scraping the old key need
updating. The new key is documented in `doc/configure.md` alongside the
new `http.302.redirection` and `http.308.redirection` success counters.

## Rollback procedure

If a regression surfaces post-2.0.0 in production, rollback to 1.1.x:

```sh
# 1. Stop the 2.0.0 binary (graceful)
systemctl stop sozu

# 2. Restore the 1.1.x binary
install -m 0755 /var/lib/sozu/sozu.1.1.1 /usr/bin/sozu

# 3. Restore the 1.1.x state file
cp /var/lib/sozu/state.1.1.x.backup.json /var/lib/sozu/state.json

# 4. Revert TOML to 1.1.x shape (see warnings below)

# 5. Restore Grafana / Prometheus dashboards from your 1.1.x JSON snapshot

# 6. Start
systemctl start sozu
```

### Known not-rollback-able

- **State files written by 2.0.0 cannot be loaded by 1.1.x.** `SaveState`
  is JSON-serialised `WorkerRequest` with serde-generated names from the
  current proto schema. There is no proven forward-compat contract for
  older serde-generated structs. **Any state mutations after the 2.0.0
  upgrade (`sozu cluster add`, `sozu certificate add`, etc.) will NOT
  survive a binary rollback.** Manually replay post-upgrade changes
  against the pre-upgrade state file backup.
- **Audit log files** written by 2.0.0's JSON sink will not parse in
  1.1.x consumers (new schema with audit-event v2 fields).
- **Access logs** with new `tls_version` / `tls_cipher` / `tls_sni` /
  `tls_alpn` / `xff_chain` fields will not parse in 1.1.x consumers.
- **Pattern-trie regex anchoring is a permanent code-level behaviour
  change.** Operators who relied on partial matching adjusted their
  regexes for 2.0.0; rolling back to 1.1.x restores partial matching
  but operators have already widened patterns — net-neutral.
- **TOML mismatch hazard**: if you added H2 ALPN to a listener at
  upgrade time WITH `buffer_size < 16393`, 1.1.x WILL NOT REJECT the
  config (it just hangs on the first H2 frame). **Revert TOML changes
  at rollback.**
- **New TOML keys** (`forwarded_for_mode`, `evict_on_queue_full`,
  `max_connections_per_ip`, `splice_pipe_capacity_bytes`,
  `basic_auth_max_credential_bytes`, `command_allowed_uids`,
  `audit_logs_target`, `audit_logs_json_target`) are silently ignored
  by 1.1.x — remove them from TOML at rollback for clean parsing.

## Supported versions after 2.0.0

Per `SECURITY.md`:

- **`2.0.x`** — current; receives security fixes.
- **`1.1.x`** — **end-of-life at 2.0.0 release.** No further security
  backports. Operators are expected to upgrade to `2.x`.
- **`1.0.x`** and earlier — unsupported.
